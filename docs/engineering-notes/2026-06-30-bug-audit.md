## Original findings (2026-06-30)

Critical Bugs

  1. backend/main.py:72 — Rules RAG is silently dead
  _rules() constructs a RulesStore and returns it without calling .load() — the only method that opens the ChromaDB collection. is_ready() is always False, so
  search_rules returns "Rulebook index is not ready" on every single call regardless of whether build_index.py has been run. The entire RAG pipeline for rules never
  works. Fix: make RulesStore lazy like HistoryStore (initialize _store on first search() call), or call rules_store.load() once in agent_lifespan.

  2. backend/tools/chargen.py:77 — Character finalization always fails when LLM uses title-case skill names
  DraftStore stores skills verbatim from the model (the docstring itself shows "Perception, Stealth, Arcana"). Character.skill_proficiencies is set[Skill] where all
  enum values are lowercase. Character(skill_proficiencies={"Perception"}) raises a Pydantic ValidationError. No try-except catches it; LangGraph surfaces it as a
  tool error; the agent retries in a loop forever. The character is never saved. Fix: {s.lower() for s in draft.get("skill_proficiencies", [])} at line 77.

  ---
  High-Severity Bugs
  
  3. backend/stores/campaign_store.py:161 — Parallel tool calls silently drop mutations
  LangGraph's ToolNode uses asyncio.gather(*coros) for all tool calls in a single AIMessage (confirmed in the installed package). Every tool independently does load 
  → mutate → save. _write_entities DELETEs all rows then re-INSERTs from the in-memory object, so the last writer wins. A model response containing
  update_character_hp + add_condition as parallel calls will silently lose one of the two changes. Fix: configure ToolNode to run tools sequentially, or add a
  version/etag column to campaigns and retry on conflict.

  4. backend/main.py:230 — add_session blocks the event loop
  _history().add_session() chains Chroma.add_documents() → OllamaEmbeddings.embed_documents() — a synchronous httpx HTTP call — directly in an async def route
  handler with no asyncio.to_thread. Every POST /session/end freezes the event loop for the duration of the embed (seconds for long summaries). Fix: await 
  asyncio.to_thread(lambda: _history().add_session(...)).

  5. backend/main.py:227 — Retry after embed failure creates a duplicate session record
  store.save(campaign) commits the session to Postgres first, then add_session runs. If Ollama is down, add_session raises, the route returns 500, and the DM
  retries. The retry finds N sessions in Postgres, computes session_number = N+1, creates a new Session UUID, and saves again — two separate session records now
  cover the same thread. Fix: wrap add_session in try-except (log but don't fail the route), and/or deduplicate on thread_id before creating a new Session.

  6. backend/agent/dm_agent.py:95 — Naive message trim can orphan a ToolMessage, breaking Ollama's message sequence
  msgs[-30:] is a raw tail slice with no awareness of tool-call group boundaries. An AIMessage with tool_calls at position 31 gets dropped while its companion
  ToolMessage at position 32 stays — leaving Ollama with [SystemMessage, ToolMessage, ...]. Ollama requires every role: tool message to follow an assistant message
  with a matching tool_call_id; it rejects or garbles the response. Reaching 32 messages takes ~8 tool-using turns, well within a normal session. Fix: trim at group
  boundaries using LangGraph's trim_messages(strategy="last", include_system=True, allow_partial=False).

  ---
  Medium-Severity Bugs
  
  7. backend/main.py:189 — Stale messages queued during disconnect are consumed by the next session's thread
  _queues[campaign_id] persists across SSE reconnections. A message submitted while the stream was down sits in the queue; when the next SSE connection arrives
  (potentially with a new thread_id from a new tab), q.get() returns the stale message and it's checkpointed into the wrong thread. Fix: encode thread_id into each
  queue entry as a (thread_id, message) tuple, and discard entries whose thread doesn't match the current stream.

  8. backend/agent/dm_agent.py:285 — Player can inject ---KEY EVENTS--- into the session summary
  Player message content is interpolated verbatim into the summarization prompt. The parser does text.split("---KEY EVENTS---", 1) — splitting on the first
  occurrence in the LLM output. A player who sends this marker as a message can cause the parser to split in the wrong place, corrupting both summary and key_events
  stored in Postgres and embedded in ChromaDB. Fix: use a randomized or sufficiently unique delimiter per invocation, or split on the last occurrence.

  9. backend/tools/chargen.py:54 — passive_perception is 3 points too high for proficient characters
  10 + wis_mod + (5 if perc_prof else 0) — the 5 is hardcoded. The proficiency bonus at level 1 is +2 (set four lines later as proficiency_bonus=2). Expertise would
  be +4. The value 5 matches no standard rule. Every Perception-proficient character gets passive_perception = 14 (WIS 14) instead of the correct 14 — wait,
  10+2+5=17 vs. 10+2+2=14. Fix: passive_perception = 10 + wis_mod + (2 if perc_prof else 0) (or reference proficiency_bonus directly).

  10. backend/tools/combat.py:112 — Monster healing is not capped at max_hp
  monster.current_hp = max(0, monster.current_hp + delta) has no upper bound. update_character_hp in party.py:92 correctly uses min(char.max_hp, ...). Fix:
  monster.current_hp = max(0, min(monster.max_hp, monster.current_hp + delta)).

  ---
  Notable non-findings: The AbilityScores.modifier floor-division handles negative scores correctly (Python // rounds toward −∞). The use_spell_slot stale-reference
  at party.py:155 is arithmetically correct (slot.max - slot.used - 1 equals the new remaining count). SQLAlchemy Core queries are fully parameterized — no SQL
  injection risk.
